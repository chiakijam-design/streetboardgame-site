# LIVE安全運用

## 1万人対応設計

- 設計上限: 1ルーム10,000人
- 本番の初期運用上限: 50人。`LIVE_OPERATIONAL_VIEWER_LIMIT`を1,000→5,000→10,000と段階的に設定して開放する
- 投票処理: 32個のDurable Objectへ参加者トークンで固定分散
- 1分割のハード上限: 400人（全体では12,800人分の余裕を確保し、公開上限は10,000人に固定）
- 10,001人目以降: `POST /api/live/games/:code/join` が `409 participant-limit-reached` を返す
- 視聴者の状態更新: WebSocket Hibernation API。接続できない場合だけ3秒ポーリングへ自動フォールバック
- スタッフ・YouTuber端末: 端末数が少ないため3秒ポーリングを継続
- 選択肢別集計: 各分割で集計し、ルーム統括へ最大約0.5秒単位で集約
- 問題締切: 32分割の確定スナップショットを取得してから結果を保存
- 視聴者本人の回答: 視聴者が属する分割だけに保存し、本人のWebSocketまたは本人トークン付き取得だけへ返す
- D1の役割: 予約、企画本体、参加者認証、確定した問題結果。通常投票はD1へ直接書き込まない
- スタッフ画面の参加者名: DOM肥大化を防ぐため先頭100人まで表示し、全参加人数は別カウンターで表示

Durable Objectsのバインディングがないローカル・退避環境では、従来方式へ戻して上限50人とする。バインディングがあっても`LIVE_OPERATIONAL_VIEWER_LIMIT`未設定時は50人に制限する。契約環境へのデプロイだけで1万人の安全性が保証されるわけではない。本番上限を引き上げる前に、同期投票・問題切替・一斉再接続を含む1,000人、5,000人、10,000人の段階負荷試験を行い、CloudflareメトリクスでCPU、エラー率、WebSocket切断率を確認する。

`wrangler.jsonc`の`LIVE_ROOM_COORDINATOR`、`LIVE_VOTE_SHARD`バインディングと、SQLiteストレージを宣言する`exports`を削除せずにデプロイする。

## 予約

- YouTube LIVEの新規保存には未来の予約日時が必須
- 最初のクローズドβでは、既存予約の開始時刻から前後20時間以内に入る開始時刻は予約不可。この値は負荷試験前に短縮しない
- 画面での空き確認後も、企画保存時にD1へ条件付きINSERTを行い競合を再確認
- 予約日時と参加上限はスタッフ用ロビーに再表示
- スタッフ用URLではライブ開始前に限り、日時変更、予約キャンセル、スタッフ用URL・本人用URLの再発行が可能
- 日時変更は、変更先をサーバー側で再確認して予約を移動できた場合だけ現在枠を解放する。競合時は現在の予約を保持する
- キャンセル時は予約枠を解放し、同じルームの全端末にはキャンセル済み画面を表示する
- 非公開URLを再発行した瞬間に旧トークンを失効する。新URLは安全な連絡手段で再共有する
- 開始時にもサーバー側で全体ロックを取得し、別のLIVEが進行中なら開始を拒否
- 最終結果まで完了した時点で全体ロックを解放。異常終了時は最長24時間で自動失効

### 予約間隔と複数開催の再判断条件

前後20時間を短縮する判断は、契約予定の本番相当Cloudflare環境で5,000人試験を完了した後に行う。次をすべて満たさなければ20時間を維持する。

- 投票・本人回答・問題切替・一斉再接続を含む通し試験が完走する
- 回答データの欠損と二重計上が0件
- LIVE API 5xx率が0.1%未満
- 意図しないWebSocket切断率が2%未満
- 投票受付のp95が2秒以内
- D1・Durable Objects・Worker CPUの主要上限に30%以上の余裕がある
- 強制終了、キャンセル、URL再発行、障害告知の運用訓練を完了する

複数同時開催は実装・開放しない。単独ルーム10,000人試験を同じ基準で安定完走し、監視と障害対応の実地確認が終わった後に、予約モデル、全体ロック、費用上限を別途設計して判断する。

## 購入用結果画像

購入用画像のプレビューには次の情報を入れる。

- YouTubeチャンネル名
- スタッフが登録したYouTuber画像。未登録時は既定キャラクター
- 予約したライブ配信日
- 視聴者が入力・変更した表示名
- 視聴者本人の正解数

購入前プレビューは、参加者トークンを検証したサーバーが540×675px・SAMPLE表示付きで生成する。YouTuber元画像は非公開R2に置き、ゲームJSON、公開API、WebSocket、視聴者のブラウザ状態へ送らない。購入権限が作成された場合だけ、サーバーが2,160×2,700px・SAMPLEなしの画像を生成し、D1権限と10分間の署名URLで購入から30日間提供する。別端末ではStripe購入メールの注文番号と決済時メールを照合して署名URLを再発行し、購入メールの平文はD1へ保存しない。

全問正解者の無料プレビュー・保存共有画像・購入済み高画質画像には、通常結果と区別できる金色テーマと王冠を使い、`【YouTubeチャンネル名】王`の称号を表示する。1問でも不正解がある場合は称号と特別装飾を表示しない。

コード上の安全境界は実装済みだが、Stripe Checkout / Connectの決済開始・Webhook、YouTube API/OAuth、R2、Cloudflare Imagesの本番資格情報は未接続である。設定手順と有料販売の許可条件は`docs/LIVE_EXTERNAL_SERVICES.md`を参照する。

応援金は結果画面の小さい「応援金を送る」ボタンから200円・500円・1,000円・3,000円を選び、Streetboardgame運営者のStripe Checkoutで決済する。応援金Checkoutは非公開R2・Cloudflare Images・画像ダウンロード署名secretに依存せず、購入履歴専用D1とStripe設定が有効なら利用できる。有料画像販売だけは画像関連の全設定を必須とする。

参加者の最終結果では、X投稿・LINE送信・プレビュー画像の保存／共有を利用できる。XとLINEには結果文と`/live`のURLを付け、画像操作は既存シリーズと同じ共通処理を使う。PCではダウンロードし、対応スマホでは画像ファイル付き共有シート、非対応スマホでは本文共有後のダウンロードへフォールバックする。
