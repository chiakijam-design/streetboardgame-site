# LIVE安全運用

## 1万人対応設計

- 設計上限: 1ルーム10,000人
- 投票処理: 32個のDurable Objectへ参加者トークンで固定分散
- 1分割のハード上限: 400人（全体では12,800人分の余裕を確保し、公開上限は10,000人に固定）
- 10,001人目以降: `POST /api/live/games/:code/join` が `409 participant-limit-reached` を返す
- 視聴者の状態更新: WebSocket Hibernation API。接続できない場合だけ3秒ポーリングへ自動フォールバック
- スタッフ・YouTuber端末: 端末数が少ないため3秒ポーリングを継続
- 選択肢別集計: 各分割で集計し、ルーム統括へ最大約0.5秒単位で集約
- 問題締切: 32分割の確定スナップショットを取得してから結果を保存
- 視聴者本人の回答: 視聴者が属する分割だけに保存し、本人のWebSocketまたは本人トークン付き取得だけへ返す
- D1の役割: 予約、企画本体、参加者認証、確定した問題結果。通常投票はD1へ直接書き込まない
- スタッフ画面の参加者名: DOM肥大化を防ぐため先頭100人まで表示し、全参加人数は別カウンターで表示

Durable Objectsのバインディングがないローカル・退避環境では、従来方式へ戻して上限50人とする。契約環境へのデプロイだけで1万人の安全性が保証されるわけではない。本番上限を公開する前に、同期投票・問題切替・一斉再接続を含む1,000人、5,000人、10,000人の段階負荷試験を行い、CloudflareメトリクスでCPU、エラー率、WebSocket切断率を確認する。

`wrangler.jsonc`の`LIVE_ROOM_COORDINATOR`、`LIVE_VOTE_SHARD`バインディングと、SQLiteストレージを宣言する`exports`を削除せずにデプロイする。

## 予約

- YouTube LIVEの新規保存には未来の予約日時が必須
- 既存予約の開始時刻から前後20時間以内に入る開始時刻は予約不可
- 画面での空き確認後も、企画保存時にD1へ条件付きINSERTを行い競合を再確認
- 予約日時と参加上限はスタッフ用ロビーに再表示
- 開始時にもサーバー側で全体ロックを取得し、別のLIVEが進行中なら開始を拒否
- 最終結果まで完了した時点で全体ロックを解放。異常終了時は最長24時間で自動失効

## 購入用結果画像

購入用画像のプレビューには次の情報を入れる。

- YouTubeチャンネル名
- スタッフが登録したYouTuber画像。未登録時は既定キャラクター
- 予約したライブ配信日
- 視聴者が入力・変更した表示名
- 視聴者本人の正解数

購入前プレビューは540×675px・SAMPLE表示付き。Stripe Checkout / Connect、購入権限、原寸画像の安全な保存先はまだ接続していないため、高画質版のダウンロードボタンは表示しない。

参加者の最終結果では、X投稿・LINE送信・プレビュー画像の保存／共有を利用できる。XとLINEには結果文と`/live`のURLを付け、画像操作は既存シリーズと同じ共通処理を使う。PCではダウンロードし、対応スマホでは画像ファイル付き共有シート、非対応スマホでは本文共有後のダウンロードへフォールバックする。
